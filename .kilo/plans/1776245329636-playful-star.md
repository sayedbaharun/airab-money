## Deployment Strategy (Direct to Repo → Railway)

### When deploying directly to Railway (no local env needed):
**Minimal local setup required** - just for verification/development:

```bash
# 1. Create local .env for testing/development only
cp .env.example .env  # configure with test values

# 2. Fix dependency issue
cd server && npm install --save-dev tsx && npm uninstall tsx

# 3. Verify build works locally (optional but recommended)
npm run build
cd server && npm run build && npm run db:push && cd ..
```

**For direct Railway deployment:**
1. Push code changes to repo
2. Railway auto-builds using nixpacks (configured in `nixpacks.toml`)
3. Railway uses its own `.env` variables (already configured)
4. Database auto-migrates via Railway's PostgreSQL setup

### Why Local Setup is Still Useful
- **Debugging**: Test locally before pushing to Railway
- **Development**: Iterate quickly without Railway deploy cycle
- **Verification**: Confirm build works before committing
- **Database testing**: Test article generation locally with mock OpenAI

## Quick Start (For Direct Railway Deployment)
